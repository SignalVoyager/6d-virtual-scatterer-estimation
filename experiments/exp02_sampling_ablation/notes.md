# Experiment Notes - Dense Sampling Ablation

## Goal
Study how different training-set generation schemes affect final model performance in the dense scene.

The target final figure is a line plot:
- x-axis: number of training samples
- y-axis: selected performance metric
- each curve: one training-set generation scheme

## Folder Contract
Self-contained experiment folder:
- `config.json`: planned configuration file for scene, sample counts, schemes, model, and metric settings
- `run_experiment.m`: planned experiment entry point
- `data/`: cached datasets, scene files, and trained response artifacts
- `outputs/logs/`: run logs and reference records
- `outputs/original/`: raw per-run metrics, raw figures, and intermediate outputs generated by code
- `outputs/final/`: paper-ready line plots and final result tables

No experiment output should be written outside this folder.

## Experimental Design
Scene:
- dense only

Sample-count grid:
- `17*4*4 = 272`
- `17*6*6 = 612`
- `17*8*8 = 1088`
- `17*10*10 = 1700`
- `17*12*12 = 2448`
- `17*14*14 = 3332`

Training-set generation schemes:
- `geom_geom`: strategic sampling around scatterers using `geom-geom`
- `rand_1tx`: choose one random TX and many random RXs using `list-rand`
- `rand_4tx`: choose four random TXs and many random RXs using `list-rand`

Sample-count interpretation:
- For `geom_geom`, the count is naturally `17 * txNumPerSc * rxNumPerSc` because the dense scene has 17 scatterers.
- For `rand_1tx`, match the same x-axis count by using `txCount = 1` and `rxNum = sampleCount`.
- For `rand_4tx`, match the same x-axis count by using `txCount = 4` and `rxNum = sampleCount / 4`.
- Example: at `17*8*8 = 1088`, `rand_4tx` uses `4` TXs and `17*8*2 = 272` RXs per TX, so `4*272 = 1088`.
- The `list-rand` implementation caps `rxNum` at the number of free RX grid points. Before final runs, verify that the highest `rand_1tx` point can actually provide `3332` RXs in the dense scene; if not, either lower the maximum count or use more TXs for that scheme.

Implementation note:
- For `rand_1tx` and `rand_4tx`, avoid repeated ray tracing for every sample count and Monte Carlo repeat.
- Generate a sufficiently large random pool once for each random scheme, then form smaller training sets by sub-sampling from that pool.
- The large pool should be generated with enough random TX/RX support to cover the largest sample count and all Monte Carlo subsets.
- This keeps the statistical comparison flexible while reducing expensive ray-tracing calls.

Monte Carlo:
- Use repeated random sub-sampling for all schemes where randomness affects the selected TX/RX pairs.
- Save every repeat's raw metrics, then aggregate mean/std or confidence intervals for plotting.

Metrics:
- Save all available evaluation metrics first.
- Choose the final y-axis metric later after inspecting the raw results.

## Expected Workflow
Planned workflow:
1. Generate or load dense-scene train/test datasets.
2. For each training-set scheme and sample count, train the selected model.
3. Evaluate on a fixed dense-scene test set.
4. Save raw metrics under `outputs/original/`.
5. Compose the paper-ready line plot under `outputs/final/`.
